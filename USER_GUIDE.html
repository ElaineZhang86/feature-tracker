Why Use Feature Context Hub?

As a designer working across multiple features at once, context is your most expensive resource. Here's what this tool is designed to protect.

Stay oriented across 5–8 features simultaneously

At any given time, you might be deep in discovery for one feature, reviewing specs for another, and answering Slack questions about a third. Without a dedicated system, the mental overhead of context-switching compounds into real productivity loss.

Schedule Overview is the antidote to Monday morning confusion. Open it once, scan every feature's status, current focus, and task progress in one view — no Jira, no Confluence, no Slack scrolling.

Context-switch without losing your place

The biggest tax on design work isn't the switching itself — it's the ramp-up time after. Every feature has a Current Focus ribbon — one sentence: what specifically needs to happen next. Update it before you switch away. When you come back, you know exactly where to pick up.

Never lose domain knowledge you worked hard to earn

Tax law nuances. API quirks. Edge cases your PM didn't know about until you surfaced them in a discovery call. This kind of knowledge is invisible in PRDs — it lives in your head, or gets buried in a Notion page nobody finds again.

The Domain Knowledge tab keeps it next to the feature it belongs to, making your future self faster, your design rationale sharper, and your call prep significantly better.

Hand off to another designer without a 2-hour catch-up call

When a feature changes hands, the outgoing designer's context usually evaporates. With Feature Context Hub, a handoff means sharing the repo. The incoming designer gets the full context tab, every discovery call with prep notes and AI summaries, every answered design question, and a clear task list showing what's blocked and why.

handshake A well-maintained feature page can replace most of a handoff meeting.

How to Set Up

1. Get the app

Clone the repository or download it as a ZIP from GitHub:

git clone https://github.com/ElaineZhang86/feature-hub.git

Then open the folder and double-click index.html to launch the app. No install, no build step.

2. Connect GitHub Sync (recommended backup)

error Don't connect to the public feature-tracker repo — your workspace data would be visible to anyone who clones it. Instead, create a private repo of your own (e.g. feature-tracker-data) and connect to that.

What you need: a GitHub account, a repository (can be private), and a Personal Access Token with repo write access.

Create a Personal Access Token

  1. Go to GitHub → Settings → Developer settings → Personal access tokens → Tokens (classic)
  2. Click Generate new token (classic)
  3. Give it a name (e.g. feature-tracker-sync)
  4. Check the repo scope
  5. Click Generate token and copy it immediately — GitHub only shows it once

Connect in Settings

  1. Open the app and click the ⚙ icon in the sidebar footer
  2. Go to GitHub Sync
  3. Fill in: Owner (your GitHub username), Repo, Branch (e.g. main), and File Path (e.g. feature-tracker-data.json)
  4. Paste your Personal Access Token and click Connect

The app verifies access and writes an initial snapshot to GitHub immediately. From then on, it automatically saves snapshots in the background — no action required.

cloud_done Once connected, your workspace follows you to any machine — open the app, reconnect with the same GitHub Sync settings, and the latest snapshot is restored into your browser automatically.
lock Your token is stored in your browser only and is never sent anywhere except directly to the GitHub API.

3. Optional — Data File backup

In addition to GitHub Sync, you can connect a local JSON file as a second backup copy — useful if you also want a file you can put in iCloud Drive or Dropbox. See Data File for details and how to restore from a backup.

Common Questions

Does the app need internet to work?

The app runs entirely in your browser — no internet required. If GitHub Sync is configured, it saves snapshots to GitHub when you're online. The app continues to work offline and saves the next time a connection is available.

Where is my data stored?

localStorage is always the live working copy — every edit lands there instantly. If GitHub Sync is configured, the app automatically saves snapshots to your GitHub repo. Your data can be restored from GitHub on any machine as long as GitHub Sync was set up before the data was lost.

How do I move to a new machine?

Open the app on your new machine, go to Settings → GitHub Sync, and enter the same owner, repo, branch, file path, and token. Click Connect — the app fetches the latest snapshot from GitHub and restores it into localStorage automatically.

Is my data safe if I accidentally clear my browser?

If GitHub Sync was configured, your data can be restored — reconnect with the same settings and the latest snapshot is loaded from GitHub. Without GitHub Sync, clearing browser data will erase your work.

How do I share my work with another person — and what's the difference between sharing a feature vs the whole workspace?

There are two different ways to share, depending on what you want to hand off:

Feature by feature (recommended for handoffs) — click the ↓ download icon on any feature header to export that one feature as a .json file. The other person clicks the ↑ upload icon next to "Features" in their sidebar to import it. Their existing features are untouched — the new feature is simply added alongside them. Works in all browsers.

Full workspace — go to Settings → Data File, click Connect file to connect a save location, then click Download to get a feature-tracker-data.json file. Send that file to the other person. They go to Settings → Data File → Load from backup and pick the file. This replaces their entire workspace with yours. Use this for a full handoff or onboarding a new designer — not for sharing a single feature. Chrome and Edge only.

GitHub Sync

Fill in your Owner, Repository, Branch, File path, and Personal Access Token. The Connect button is disabled until all three required fields (owner, repo, token) are filled. Click Connect — the app verifies credentials, writes your data to GitHub, and collapses the form.

When connected, the status row shows a green dot with your repo name and the last save time. A Disconnect button appears in the section header. Clicking it shows a confirmation modal before disconnecting — your field values are preserved so you can reconnect without retyping.

Data File

The Data File section has two actions:

Connect file / Download — when no file is connected, clicking Connect file prompts you to pick a save location. The app automatically writes updated snapshots to that file. Once connected, the button becomes Download — clicking it downloads the current workspace as feature-tracker-data.json. The status row shows the connected filename and file size.

Load from backup — pick any .json backup file to restore your full workspace. This replaces everything in your current session. Useful after clearing the browser, moving to a new machine, or receiving a workspace export from a teammate.

warning The File System Access API (Connect file / Download / Load from backup) is only available in Chrome and Edge. It will not work in Safari or Firefox.

Adding a New Feature

Click the + button next to "Features" in the sidebar. Fill in:

  • Title, icon, status, customer, due date
  • Summary, current focus
  • Team members

Exporting a feature

Click the download icon (↓) in the feature header to export the feature as a .json file. The file includes the feature definition (title, status, customer, team) and all its data — calls, domain knowledge, notes, tasks, Q&A, design files, and spec links.

Share this file with anyone who uses Feature Hub to give them a full copy of the feature with all its context intact.

Importing a feature

Click the upload icon (↑) next to "Features" in the sidebar. Pick a .json file exported from Feature Hub. The feature is added as a new custom feature and you are taken straight to it. All data — calls, domain knowledge, notes, tasks, Q&A — is restored exactly as it was exported.

handshake Export + Import is the fastest way to hand off a feature to another designer. Export takes one click. The incoming designer gets everything — context, research, calls, decisions — without a catch-up meeting.

Feature Detail Page

Each feature has a header with an editable status dropdown and due date picker, then a Current Focus ribbon you can update at any time.

Below that are tabs, each covering a different dimension of the feature.

Context

A reference card showing the core feature brief:

  • Summary — one-paragraph problem statement
  • Key Decisions — confirmed design/product decisions
  • Constraints — known limitations and out-of-scope items
  • Links — Jira tickets, PRDs, Gong calls, design files
  • Team — PM, Eng, stakeholders with their roles
check_circle Use this tab first whenever you're re-orienting to a feature after time away. It replaces the need to re-read the PRD or scroll Slack history.

Domain Knowledge

A personal notebook for context you have built up over time — edge cases, mental models, integration quirks, anything not in the PRD.

  • Click + Add to create a new entry
  • Entries are collapsible, editable, and deletable
  • Title autosaves when you click away
handshake Handoff tip: A well-filled Domain Knowledge tab is the single biggest time-saver for an incoming designer. If you had to figure something out, write it here.

Feature Brief

Structured background context — architecture decisions, data models, domain deep-dives.

  • Click + Add to create a new entry
  • Same collapsible / editable / deletable pattern as Domain Knowledge

Calls

Log all calls related to a feature — customer discovery, internal syncs, design reviews.

Creating a call

Click + Add and choose Customer Call or Internal Call. The call opens immediately in edit mode — no modal or popup. Fill in the fields directly:

  • Call Name — primary identifier
  • Type — Customer or Internal
  • Category — Customer Discovery, Design Review, Internal Sync, etc.
  • Status — Scheduled / Completed / Action Items Pending
  • Date, Customer, Attendees, Notes Link
  • External Resources — attach slides, recordings, or docs with a label and URL

Click the edit icon (pencil) on any existing call card to switch it to edit mode. When editing, click Save to confirm your changes or Discard to revert them. Discarding a brand new call removes it entirely.

Inside each call — three tabs

  • General Info — call details, attendees, resources
  • Call Prep — Research Questions, Interviewee Background, Warm-up Questions, Topic Questions, Look For / Listen For. Use Copy prep prompt to Claude.ai to generate AI-assisted prep.
  • Post Meeting — paste the transcript; generate an AI summary with Claude
tips_and_updates Context-switch tip: Create the call card as Scheduled before the call happens. Fill in prep. Paste the transcript after. Everything stays in one place and compounds over time.

Q&A

A question bank with tracked answers.

  • Add questions with + Add
  • Answers support markdown
  • Mark questions as Resolved when answered; filter by open / resolved
handshake Handoff tip: The Q&A tab is often the best signal of how thoroughly a feature has been explored. An incoming designer can read through it and immediately understand what was unknown, what got answered, and what's still open.

Design Spec

Links and embeds for your design spec.

  • Add spec links with + Add
  • Edit or delete existing specs

Design Files

Links to Figma files, prototypes, or other design assets.

  • Add links with + Add
  • Links open in a new tab

Notes

A free-form scratchpad for anything that doesn't fit elsewhere — raw thoughts, reminders, Slack messages to follow up on. Autosaves as you type.

To Do

A unified task list for all work states.

StatusMeaning

DoneCompleted

  • Edit a task — click the task text to edit it inline; press Enter or Escape to save, or click away

Sections

Group tasks with section headers to organize your list.

  • Click + Section to add a new section header
  • Rename a section — click the section title to edit it inline
  • Delete a section — click the trash icon on the section header (tasks inside are not deleted)
  • Drag to reorder — drag any task above or below another task, or drop it onto a section header to place it directly below that section
handshake Handoff tip: Postponed tasks are especially valuable during handoffs — they tell the incoming designer what's blocked and why, so nothing falls through the cracks.

Getting Updates

If you cloned the repo, you can pull in the latest version of the app at any time:

  1. Open your terminal and navigate to the feature-tracker folder
  2. Run git pull
  3. Refresh the page in your browser

Your data is stored in localStorage and is not affected by pulling updates. If GitHub Sync or a Data File is connected, your snapshots are also unaffected.

Deleted Features

Deleted features appear in the Deleted section at the bottom of the sidebar.

  • Restore — brings the feature back
  • Delete forever — permanent removal (confirmation required)

Tips

  • Set up GitHub Sync first — open Settings → GitHub Sync and connect a repo before you start adding features. Takes 2 minutes once, then runs silently in the background forever.
  • Start every week on Schedule Overview — scan current focus and progress, update anything stale before diving in
  • Use Current Focus as a daily intent — one sentence: what specifically needs to happen on this feature today
  • Log calls even before they happen — create as Scheduled, fill in prep, paste the transcript after
  • Postpone, don't delete — if a task is blocked, postpone it with a reason so the context survives a handoff
  • Use markdown in Q&A answers — tables work well for comparing options; headers make long answers scannable
  • Domain Knowledge is for things you learned, not things you were told — if you had to figure it out, write it down so you never have to again
  • A well-maintained feature page can replace most of a handoff meeting — the more you put in, the faster someone else can pick it up
  • Export before you hand off — share a feature's full context with a teammate as a single file
